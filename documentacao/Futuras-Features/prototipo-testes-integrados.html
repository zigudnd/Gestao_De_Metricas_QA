Análise UX — Antes vs Proposta

Comparação da experiência atual com a proposta de melhoria

❌ Problemas Atuais

1. Suite genérica "iOS" O nome da suite não indica qual squad é responsável. O usuário precisa renomear manualmente.
2. Sem indicação de conformidade Não há como saber se o squad já cadastrou testes suficientes. A suite mostra "1 func. · 1 testes" sem contexto de PRs.
3. Campo "Nome da Funcionalidade" ocupa muita área O label "NOME DA FUNCIONALIDADE" + input grande desperdiça espaço vertical. O nome pode ser inline no header.
4. Cenários Gherkin expostos por padrão O textarea de Gherkin ocupa 50% do espaço da feature. A maioria dos testes integrados não usa Gherkin — são testes manuais.
5. Sem contexto de PRs aprovados O QA não vê quantos PRs cada squad subiu. Precisa sair da sprint e ir para Gestão de PRs para verificar.
6. Suites criadas manualmente O usuário precisa criar suites, renomear para o nome do squad, e depois associar features. 3+ cliques antes de começar a cadastrar testes.
7. Select de squad na feature Para sprints integradas, ter um select de squad em cada feature é redundante quando a suite já representa o squad.

✅ Melhorias Propostas

1. Suites = Squads (auto-criadas) Ao abrir a sprint integrada, suites são criadas automaticamente com o nome dos squads que têm PRs aprovados. Zero configuração.
2. Badge de conformidade por suite Cada suite/squad mostra "✅ Conforme" ou "⏳ Pendente" no header. Visível sem expandir.
3. Feature name inline no header O nome da funcionalidade é um input inline no header da feature, sem label separado. Economiza ~40px de altura.
4. Gherkin colapsado por padrão O textarea de Gherkin fica oculto e aparece ao clicar em "Expandir Gherkin". Para testes manuais, economiza espaço.
5. Painel "Testes Integrados por Squad" Acima das suites, um painel mostra PRs aprovados × testes cadastrados por squad, com barra de progresso e badges.
6. Empty state orientativo Squads sem testes mostram mensagem clara: "Este squad tem X PRs aprovados e precisa registrar testes integrados."
7. squadId automático Features herdam o squadId da suite automaticamente. Sem select redundante. O QA só precisa: expandir squad → criar feature → adicionar test cases.
📋 Fluxo simplificado (3 passos)
1. Criar sprint integrada vinculada à release → suites auto-criadas por squad
2. Expandir squad → Adicionar Funcionalidade → nomear
3. Adicionar Caso de Teste → preencher nome e complexidade → pronto
